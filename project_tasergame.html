TASER GAME

Combining paintball and laser tag — real consequences, zero mess.

FreeRTOS  ·  ESP-NOW  ·  Custom PCB  ·  Li-ion

← Back to Projects

The Problem

The gap between Laser Tag and Paintball is too large. There needs to be a middleground where the risk of getting hurt is zero, but the risk of feeling pain is real. Paintball is very immersive because the consequences of getting hit are tangible. However, getting hit in the neck leaves marks for days — the whole thing must be performed in a dedicated dirty space, and your shoes might as well go straight into the washing machine when you return.

Another major drawback of paintball is the complete absence of game statistics. Nobody knows who hit who, or who performed best.

The Hypothesis

The solution seemed straightforward: build a common IR LED + receiver system with an ESP32 and add a shock circuit reverse-engineered from a generic shock collar. I used a custom hobby laser tag system built by a peer as the baseline, and bought similar components for a first 2-pistol prototype to test the IR communication.

Iteration Log

V1 — First 10-Pistol Set

After the two-pistol prototype worked, I assembled 8 more and added a master ESP32 to periodically poll each pistol for its HitData, assembling a single true scoreboard based only on IR hits. Lost WiFi packets never become a problem — if a pistol is out of master range, the scoreboard simply is not updated until it returns. The IR hit is always the truth. HP is ALWAYS lost regardless of whether the hit confirmation reaches the shooter.

The first 10-pistol set used hardcoded master/slave MAC addresses for simplified code. While it successfully demonstrated the core mechanics, it is not scalable — replacing one device means reprogramming all others.

I brought the first set to the laser tag venue VRGame in Nørresundby. The owner was very impressed and noted that the shock circuit is a unique and valuable proposition. He also described the everyday friction with their existing system — manually turning on and connecting each pistol. He stated that for him to seriously consider a system like mine, the pistols would need to be physically larger and more robust.

V2 — Scalability and Durability

The second iteration introduced a non-hardcoded master/slave architecture with dynamic pairing — the master broadcasts and polls slaves by discovered MAC address, managing active players without hardcoded values. The focus was entirely on scalability and durability: fewer components, less manual labour, and a pistol that can survive violent use.

  • Switched from triple-LED indicators to the single inbuilt NeoPixel on the microcontroller
  • Switched from the bulky TDisplay ESP32-S3 to the smaller Waveshare ESP32-S3 variant
  • Switched to a larger toy pistol housing, allowing all components to fit securely inside
  • Switched from flimsy cord to comfortable and durable watch bands for the shock circuit
  • Switched from manually soldered components to a custom PCB for all switching circuits

Technical Build

Hardware Architecture

  • Topology: Master/Slave with peer-to-peer elements — 1 Master Base Station, currently 10 Slave pistols
  • Slave MCU: Waveshare ESP32-S3 1.47" LCD boards
  • Master MCU: ESP32 WROOM with external antenna
  • Communication: ESP-NOW. Master polls sequentially for ScoreboardData. Slaves respond with running totals (HitData). Hit confirmations sent peer-to-peer between slaves

Electronics & PCB

  • IR system: VSLY5940 IR LED controlled by N-channel MOSFET. 3× TSOP32338 receivers per gun
  • Feedback: Existing toy gun speaker and vibration motor wired to switching circuits on the added PCB
  • Shock module: 4V to 1800V converter controlled via N-channel MOSFET for micro-ampere short shocks
  • Power: 18650 Li-ion cell using the ESP32's inbuilt charging circuit. Power bank for base station
  • Display: 1.47" TFT display providing real-time player status

Embedded Software — RTOS

  • Protocol: NEC protocol (38kHz carrier, 67.5ms packet duration). Packets include PlayerID, Damage, and an 8-bit Timestamp for duplicate hit rejection
  • Task architecture: FreeRTOS with dual-core task division — non-time-critical tasks on Core 0, all network tasks on Core 1
  • Data integrity: Slaves maintain authoritative running totals. Master aggregates for eventual consistency. Strict MUTEX implementation prevents shared resource corruption

Tablet UI

A tablet connects directly to the Master's web server with no app installation required. The interface provides game configuration (modes, team count, duration), a real-time global scoreboard during play showing kills, damage dealt, and team scores, and a final results screen highlighting top performers.

User Validation

The V1 set was tested with a gymnastics team to evaluate shock magnitude, ease of use, and game mode logic.

  • Shock magnitude: Perceived as too high at the lowest setting, with no significant difference between medium and high. Linear duration parameters do not equate to linear pain perception. Fix: shorten the low-setting pulse, keep medium, extend high to differentiate more clearly
  • Usability: Game mode logic was intuitive and worked without explanation. Tablet UI was easy to navigate

Lessons Learned

Manual assembly and retrofitting took much more time than anticipated. Many hours can be saved by designing a PCB earlier and planning for the least possible physical change when retrofitting an existing product — the V2 PCB proved this dramatically.
